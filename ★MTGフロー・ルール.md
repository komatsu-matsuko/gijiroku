# ★ MTGフロー・ルール

**※ 忘れないように：MTG開始〜議事録作成の流れはここにまとめてあります。今後もルール的な内容はこのファイルに追記します。**

**※ 社長として「次回MTGをどう進めるか」の手引きが欲しいときは、`設定/社長用_MTG進め方手引き.md` を開く。**

**※ 社員が増えた・制度が変わったときなど、**どのファイルを更新すればよいか**は `議事録_保守・更新ガイド.md` を見る（誰が担当しても同じように議事録の頭脳として使えるための一覧）。**

**※ 週次MTGの進捗確認・事前準備を「効果的かつ精度高く」やりたいときは、`設定/週次MTG_進捗・事前準備ルール.md` を開く（Async First・Pre-Read・Notion/Slack・GASの運用）。**

## ★ MTG開始 〜 議事録作成（手順）

| 順 | あなたが送る | その後やること |
|----|--------------|----------------|
| 1 | **MTG開始** | AIが**「何のMTGですか？」**と聞く |
| 2 | 答える（例：**神の手**／幹部MTG） | 「神の手」なら `設定/MTG種別/神の手.md` を参照し、MTG種類・日付・出席者を聞く。それ以外は従来どおり参照情報・日付・会議名・出席者を聞く |
| 3 | それに答える | （参照情報の有無、日付・会議名・出席者など） |
| 4 | — | AIが前提条件をまとめ、「MTGを開始しますか？確認」と返す |
| 5 | **y** | AIが「音声・追加情報を入力し、不足情報の補足や内容の整理を一緒にできます。準備ができたら「MTG終了」と送ってください」と案内 |
| 6 | 音声（または文字起こし）と追加情報を入力 | **MTG終了まで**：不足情報の補足・内容の整理をAIと一緒に行う（議事録はまだ作らない） |
| 7 | **MTG終了** | AIが**議事録・Todo等の作成を開始**（①〜⑥＋不明点） |
| 8 | （内容を確認） | **手動でSlackに送る**（社長用は 設定/slack.md の D01KDC0NN7M） |

### 「何のMTGか」と神の手

- **MTG開始**でAIが最初に聞く「何のMTGですか？」に **「神の手」** と答えると、ラーメン居酒屋・神の手用の定義が読み込まれる。
- 神の手は**チーム構成が複雑**なため、MTGでは**「誰が何をやるか」の整理**を行う。**音声（または文字起こし）を渡したら**、その内容をもとに **MTG結果をまとめ**、**誰が何をやるべきか** を整理して出力する。
- **締切・担当者など不明な情報**がある場合は、推測せず**必ず社長に聞く**。AIは「以下の点を教えてください」として不明な項目を箇条書きする。
- 神の手の**チーム構成・MTGの種類・スケジュールの考え方・伝え方**は **`設定/MTG種別/神の手.md`** にまとめてある（新メニュー開発／新ツール導入／ルール変更／運営定例／メディア・発信 など）。議事録の会議名・出席者・Todoの担当はこの定義に沿って整理する。

## MTGで使うツール・準備で開くもの

- **文字起こし**： [Notta](https://app.notta.ai/7436970472509075456/dashboard) を使用する。
- **MTG準備**：**「MTG準備」** と送ると、AIが前回の議事録・意思決定ログ・次回MTG設定を見返し、**社長が準備できているかの振り返り**を出力する（社長の宿題・次回で確認することのチェックリスト）。Slack・Cometの起動は行わない。Notta・Notion・Slackは必要に応じて手動で開く。

## 「議事録を作成」で出るもの

**MTG終了時は、まず「決まったこと」と「誰が何をすべきか」がすぐ分かる一覧を冒頭に出す。**

① **決まったこと・誰が何をすべきか**（冒頭）決まった事項の箇条書き＋担当者別のやること・期限の一覧（表または箇条書き）  
② 議事録＋会議参加者への指示（Todo）  
③ 意思決定ログ  
④ 次回の会議前ブリーフィング  
⑤ 役員へのFB内容  
⑥ 経営アドバイザー所見  
⑦ **CEO思考レビュー（MTG振り返り）**（6項目×5段階・合計点・総合評価・良かった判断・改善すべき判断・次回意識すること）  
⑧ 社長のSlack送信用（D01KDC0NN7M）  
⑨ **送信用テキスト**（必ず作成）`議事録/YYYY-MM-DD_会議名_送信用テキスト.md` に保存。フォーマットは既存の `*_送信用テキスト.md`（幹部MTG等）に準拠（■MTG議事録／会議概要／テーマ別【1.〜】／次回MTGまでの宿題／以上です）  
（該当時）「以下の点が不明瞭です。お答えください。」の箇条書き  

**名前の最適化**：議事録・送信用テキストに登場する名前は、**`楽しい仲間達/`** を参照し、同フォルダ内の表記（姓名・呼び方）に合わせて統一・修正する。該当する人物がいない場合は文字起こしの誤りとみなし、文脈から該当しそうなスタッフ名に修正する。

## このワークフロー実行に最適な拡張機能

Cursor の拡張機能パネル（**Cmd + Shift + X**）で、次のように検索してインストールする。IDで入れない場合は「名前」で検索する。

### 録音・文字起こし（音声 → テキスト）

| 検索キーワード | 拡張の例 | 用途 | 備考 |
|----------------|----------|------|------|
| **Whisper** | Speech to Text with Whisper / Cursor for Speech / Whisper Voice Input / Yap | 録音→文字起こし。日本語精度が高い。チャットに送れるものは議事録作成まで一気に使える | 多くは OpenAI API キーまたはローカルモデル要。FFmpeg 要のことが多い |
| **Speech** | VS Code Speech（Microsoft） | 録音→文字起こし。ローカル・無料。API不要 | 設定で `accessibility.voice.speechLanguage` を **ja-JP** に |
| **VoxPilot** | VoxPilot | オンデバイスで録音→文字起こし。APIキー不要 | Cmd+Alt+I でカーソル位置に挿入。英語向きが強い場合あり |

- **推奨**：議事録用は **Whisper 系**（日本語・精度）か **VS Code Speech**（無料・ローカル）のどちらか。Whisper 系なら「チャットに送信」できるものを選ぶと MTG終了後の流れがスムーズ。

### Slack 送信（議事録・Todo を送る）

| 検索キーワード | 拡張の例 | 用途 | 備考 |
|----------------|----------|------|------|
| **Slack** / **vscode2slack** | VSCode 2 Slack（siesing） | エディタから選択テキストやファイルを Slack のチャンネル・DM に送信 | ワークスペースのトークン設定が必要。議事録・Todo を確認後にそのまま送れる |

- 現状は「内容を確認して**手動で**Slackに送る」想定。拡張を入れると、出力を選択して Cmd+U M などで送信でき、手間が減る。

### インストール手順

1. **Cmd + Shift + X** で拡張を開く  
2. 上記の**検索キーワード**で検索  
3. 目的に合う拡張を選んで **インストール**  
4. 音声系：必要なら FFmpeg（`brew install ffmpeg`）・OpenAI API キー・言語を ja-JP に設定  
5. Slack 系：設定で Slack トークンやワークスペースを指定  

ワークスペースの推奨（.vscode/extensions.json）にも同じ拡張を登録してある。一覧に「推奨」が出たらそこから入れてもよい。

## ★ 役割分担（6役）

| # | 役割 | 主な担当 |
|---|------|----------|
| 1 | **オーケストレーター** | MTG開始／y／MTG終了の案内、全体の取りまとめ |
| 2 | **記録係** | 議事録の作成 |
| 3 | **Todo係** | メンバー別のSlack用Todo文面 |
| 4 | **ナレッジ係** | 議事録・決定事項のナレッジ化 |
| 5 | **経営アドバイザー** | 幹部の未熟な点・改善点の所見 |
| 6 | **役員FB担当** | 評価制度に基づく役員向けFB |

※ 実際の実行は1つのAIが6役分の出力をまとめて行う。

---

## AIを賢くするために渡すとよい情報

各役割の出力の質を上げるために、次の情報を事前に用意し、必要に応じてチャットで共有するか、フォルダに置いて @ で参照させるとよい。

| 役割 | 渡すとよい情報 | 例 |
|------|----------------|-----|
| **記録係** | 会議の型・用語 | アジェンダのテンプレ、社内略語・プロジェクト名、出席者（役職・名前）の一覧 |
| **Todo係** | メンバー情報・Slackのルール | 幹部・役員の名前・役職、Todoの書き方の偏好（締切の表記など）、担当の割り当て方 |
| **ナレッジ係** | 分類・キーワードのルール | 決定事項のカテゴリ、検索で使うキーワードの傾向、過去ナレッジのフォルダ構成 |
| **経営アドバイザー** | 経営の前提・期待 | 重視している価値観、リーダーシップの期待、戦略の方向性、「未熟」の観点（例：意思決定の速さ・共有の質） |
| **役員FB担当** | **評価制度・ルール** | 評価項目・基準、等級・ランク、FBのフォーマット、観点（コンピテンシーなど）、評価期間 |
| **オーケストレーター** | 運用の前提 | MTGの頻度、次回日時の決め方、特別なフロー（あれば） |

**置き場所の例**：`参照情報/` に置き、議事録作成前に「@参照情報/ を参照して」と送るか、下記「より高度にする方法」で自動参照させる。

## ファイルを渡すと、より高度にする方法

以下のようにすると、渡したファイルを確実に効かせられ、出力の質が上がる。

### 1. ファイル名と役割を固定する

| ファイル（参照情報/ 内） | 主に効く役割 |
|--------------------------|--------------|
| `評価制度・FBルール.md` | 役員FB（フォーマット等） |
| `役員_評価制度.md` | 役員FB（月1回評価・6項目と配点）。FB内容は月末に一緒に考える |
| `メンバー一覧.md` | 記録係・Todo係 |
| `会議の型・用語.md` | 記録係 |
| `一般社員_評価制度.md` | 一般社員が参加する場合のFB・次のタスク（6項目） |
| `経営の前提・アドバイザー観点.md` | 経営アドバイザー |
| `社長用人事評価指標.md` | 社長の人事評価・振り返り（6項目・5段階。MTG評価とは別） |

この名前で置いておくと、AIが「どのファイルを読むか」を迷わず、議事録作成時に自動で参照するルールになっている。

### 2. 中身は見出しでブロック分けする

同じファイル内で「どの段落が何に使うか」が分かると、抜き出しが安定する。

- **評価制度・FBルール.md** の例：`## 評価項目` `## 等級・基準` `## FBの観点（コンピテンシー等）` `## フォーマット例`
- **メンバー一覧.md** の例：`## 幹部` `## 役員` の下に名前・役職・担当を箇条書き
- **経営の前提・アドバイザー観点.md** の例：`## 重視する価値観` `## リーダーシップの期待` `## 所見で見る観点（未熟の定義）`

こうしておくと、役員FB・Todoの担当者・経営アドバイザー所見で、それぞれ同じ観点が毎回使われる。

### 3. 議事録作成時の「自動参照」を利用する

「議事録を作成」と送る時点で、**参照情報/** 内の上記ファイルがあれば、AIが読みに行き、評価制度・メンバー・用語・アドバイザー観点に反映する。毎回 @ で指定しなくてもよい（必要なら「@参照情報/評価制度・FBルール.md を優先して」などと指定可能）。

### 4. 長い規定は「要約＋参照」にする

評価制度や就業規則が長文のときは、**参照情報用に要約版**（評価項目・等級・FBの観点だけ）を 1 ファイルにまとめ、詳細は「正式版は ○○ を参照」と一行だけ書く。AIは要約版を主に使い、必要なら詳細を読む。

### 5. 更新したら「最新の ○○ を使って」と伝える

ファイルを差し替えたあと、「次回の議事録作成から、参照情報の最新版を使って」と一度送っておくと、以後その内容が優先される。

## 送信用議事録のフォーマット（濱村・笹川などに送る用）

議事録をSlack等にそのまま貼り付けて送るための「送信用テキスト」は、次の改行・記号のクセに従う。議事録のまとめ方（何を書くか）はこれまで通り。学習するのは改行や記号の使い方のみ。

**冒頭**
・1行目：■MTG議事録
・2行目：お疲れ様です。
・空行
・3/10の幹部MTGの議事録を共有します。（次回MTGの案内はここでは書かず、会議概要にまとめる）
・空行

**見出し**
・見出しは【】で囲む（例：【会議概要】【1. 本社事務部門 賞与制度（濱村案）】）。**##** は使わない。
・セクションとセクションの間は空行1行。

**本文の記号・改行**
・「決定」「方針」など小見出しは、その行だけにして、次の行から中身を書く。中身が ①②③ のときは **・** をつけず、改行だけで並べる。
・方針の下など、複数項目を並べるときは **・**（中黒）で始める。**-** は使わない。
・キー：値（日付：、出席者：など）は改行で並べるだけ。・はつけない。
・太字 **\*\*** は使わない。プレーンテキストのみ。

**宿題ブロック**
・見出しは【次回MTGまでの宿題（3/17まで）】とする（「お願いしたいこと」ではなく「宿題」）。
・担当者名は「油谷」「濱村さん」「笹川さん」のように表記。油谷は「（社長）」をつけない。
・前の行の続きを示すときは「・↑」を使ってもよい。

**結び**
・「以上です。」のあと改行し、次回MTGの案内を1文で書く。
・最後は「相違点、不明点があればお知らせください。」とする。

## 楽しい仲間達（全スタッフ）と文字起こしでの名前

**楽しい仲間達** フォルダには弊社の全スタッフの情報を置く。文字起こし・議事録作成では次の運用とする。

・**名前の参照**：文字起こしなどで名前が出てきた場合は、このフォルダ内の人物であることを前提に扱う。
・**名前の誤り修正**：内容から明らかにスタッフの話なのに、楽しい仲間達の一覧に該当する名前がない場合は、**文字起こしの誤り**とみなす。文脈から該当しそうな弊社スタッフの名前に修正して議事録等に反映する。**今後のMTGでも同じ運用**とする。
・**個人の特性・配慮**：文字起こしやMTGの内容から、スタッフ個人について気をつけることやアドバイス可能なことが分かった場合は、**アドバイス可能なナレッジ**として `ナレッジ/` に保管する（例：スタッフ_特性・アドバイス.md）。議事録作成時やFB・所見で参照する。
